# Contrato S0.1 — Snapshot de lista por sesión

## Problema

`/borrar N`, `/cancelar_recordatorio N`, `/editar N`, "listo N", el flujo de
prioridad — todos toman un índice 1-based que el usuario ve en una lista. Dos
fallos:

1. **Objeto equivocado.** `/borrar` opera sobre microtasks, `/cancelar_recordatorio`
   sobre reminders. El usuario percibe UNA lista. Si ve `/recordatorios` y dice
   `/borrar 1`, el bot borra la microtask #1 (otra cosa) y **confirma éxito falso**.
2. **Lista stale.** El índice se resuelve contra la lista ACTUAL, no contra la
   que el usuario vio. Si la lista cambió entre el listado y el comando, borra
   el item equivocado.

## Solución: snapshot de la última lista mostrada

Cuando el bot muestra una lista numerada (`/recordatorios`, `/focus`), guarda en
la **sesión** un snapshot:

```typescript
interface ListSnapshot {
  kind: 'reminders' | 'microtasks';
  shownAt: string;            // ISO — para TTL futuro
  items: Array<{
    position: number;         // 1-based, lo que el usuario vio
    refId: string;            // id ESTABLE del item
    text: string;             // para confirmación
  }>;
}
```

El snapshot vive en `ISessionStore` (genérico, agnóstico — cualquier dominio con
listas numeradas lo necesita). NO en `IAdhdCoachStore`.

## Resolución de índice

Cuando el usuario manda un comando con índice:

1. Leer el snapshot.
2. Si existe y la posición está → usar `{kind, refId, text}` del snapshot.
3. Ejecutar la acción sobre `refId` con el método `*ById` correcto según `kind`.
4. Confirmar mostrando texto + tipo: `"Borrado: 'X' (recordatorio)"` — el usuario
   ve de inmediato si fue lo correcto.
5. Si el `refId` ya no existe (snapshot stale): `"'X' ya no está. Mira la lista
   actual."`
6. Si NO hay snapshot:
   - `/cancelar_recordatorio N` → reminders (lista actual).
   - `/borrar N` → microtasks (lista actual).
   - `borra/cancela N` (NL sin sustantivo) → inferencia: microtasks si hay,
     si no reminders.

## Ids estables para microtasks

Hoy `getMicroTasks` devuelve `id = String(posición)` — NO estable. Se cambia:

- **Memoria**: counter `microTaskSeq` que nunca se reusa. `id = String(++seq)`.
- **Postgres**: `id = String(dbId)` (el SERIAL ya es estable).

Los reminders ya tienen id estable.

## Cambios de API

### `ISessionStore` (+3)
```typescript
setLastList(userId: string, snapshot: ListSnapshot): Promise<void>;
getLastList(userId: string): Promise<ListSnapshot | null>;
clearLastList(userId: string): Promise<void>;
```

### `IAdhdCoachStore` — `*ByIndex` → `*ById`
```typescript
// reemplazan a deleteMicroTaskByIndex / editMicroTaskByIndex / setMicroTaskPriority
deleteMicroTaskById(userId, id): Promise<string | null>;
editMicroTaskById(userId, id, newText): Promise<string | null>;
setMicroTaskPriorityById(userId, id, priority): Promise<string | null>;
// nuevo: cancelar reminder por id estable
cancelReminderById(userId, reminderId): Promise<string | null>;
// getMicroTasks: el campo `id` ahora es ESTABLE (no posición)
```

`cancelReminderByIndex` se mantiene (lo usa el fallback sin snapshot).

## Handlers afectados

- `listTodayFocus` → guarda snapshot `kind='microtasks'`.
- `listReminders` → guarda snapshot `kind='reminders'`.
- `deleteMicroTask`, `editMicroTask`, `completeMicroTask`, `cancelReminder` →
  resuelven posición vía snapshot, ejecutan por id, confirman con tipo.
- `prioritizeTasksStart` / `prioritizeStep` → el draft guarda **ids estables**
  en vez de índices (el flujo es self-contained, no usa snapshot pero sí gana
  robustez con ids estables).

## Compatibilidad v4

- El snapshot es aditivo en `ISessionStore` (genérico). No engorda el dominio.
- Postgres: el snapshot se persiste en `sessions` con `type='last_list'` y JSON
  en `data`. Cero migración.
- `cancelReminderByIndex` se conserva → no rompe nada existente.

## Tests

1. `/recordatorios` luego `/borrar 1` → cancela el recordatorio #1 (no microtask).
2. `/focus` luego `/borrar 1` → borra la microtask #1.
3. Confirmación muestra el texto + tipo correcto.
4. Snapshot stale (item ya borrado) → mensaje claro, no crash.
5. Sin snapshot → fallback por tipo de comando.
6. Ids de microtask estables tras borrados intermedios.
7. Flujo de prioridad sigue funcionando con ids estables.
